Detailed Scenarios

Scenario 1: One Person, Two Branches, Different Permissions
The Situation
Sometimes a person works across two branches but plays different roles in each. In one branch, they are an active operator. In the other, they only need to keep an eye on things without making changes.
Real-World Example
	•	Ramesh is a regional operations controller.
	•	In the South branch, he manages day-to-day operations — creating trips, editing details, and closing journeys.
	•	In the West branch, he only monitors performance dashboards and journey status. He should not be able to change anything there.
How It Is Set Up
	•	South branch → Full access (create, read, edit, delete).
	•	West branch → View-only access.
What Should Happen
	•	Ramesh can create, edit, and delete journeys in the South branch.
	•	He can view journeys in the West branch but cannot change or delete anything.
	•	If he tries to edit a West branch journey, the system blocks him.

Scenario 2: Two Teams in the Same Branch, Separate Data
The Situation
Two different teams operate out of the same branch, but they handle completely different types of goods. Each team should only see their own work and not the other’s.
Real-World Example
	•	Mumbai branch has two teams: Team A handles FMCG goods (like soaps and packaged food), and Team B handles Cement.
	•	Both teams sit in the same office and may even share some vehicles or warehouses.
	•	But their transaction data must be kept separate — Team A should never see Cement journeys, and Team B should never see FMCG journeys.
How It Is Set Up
	•	User 1 (Team A) is tagged with the FMCG label.
	•	User 2 (Team B) is tagged with the Cement label.
	•	Some shared resources (like vehicles) may appear for both, but journey records are isolated.
What Should Happen
	•	User 1 logs in and only sees FMCG journeys.
	•	User 2 logs in and only sees Cement journeys.
	•	Neither user can see, search for, or access the other team’s data.
	•	Shared master data (vehicles, warehouses) is available to both where needed.

Scenario 3: Business Head vs Regional Head (Hierarchy Within a Branch)
The Situation
A branch may be divided into multiple regions. The business head needs to see everything across all regions but should not make changes. Each regional head should fully manage their own region only.
Real-World Example
	•	SPD branch is divided into four regions: North, South, East, and West.
	•	Mr. Sharma is the Business Head for SPD. He needs a bird’s-eye view of all four regions.
	•	Ms. Priya is the Regional Head for SPD North. She manages all North region operations.
How It Is Set Up
	•	Mr. Sharma gets an SPD_ALL tag → View-only access across all regions.
	•	Ms. Priya gets an SPD_NORTH tag → Full access to North region only.
	•	Similar tags for South, East, and West regional heads.
What Should Happen
	•	Mr. Sharma can see all journeys across all four regions but cannot create, edit, or delete any journey.
	•	Ms. Priya can create, edit, and delete journeys in the North region only.
	•	Ms. Priya cannot see or access South, East, or West data.
	•	Clear boundaries are maintained between oversight and operational roles.

Scenario 4: Cross-Branch Regional Leader
The Situation
A senior leader is responsible for a specific region but that region spans across multiple branches. They need to see their region’s data regardless of which branch it belongs to.
Real-World Example
	•	Mr. Vikram is the North Regional Director.
	•	His region’s operations run through three branches: SPD, TMCV, and DEF.
	•	He needs to see all North region journeys from all three branches in one view.
How It Is Set Up
	•	Mr. Vikram is tagged with “North Region” access across SPD, TMCV, and DEF branches.
What Should Happen
	•	Mr. Vikram can filter and view all North region journeys across all three branches.
	•	He cannot see South, East, or West region data from any branch.
	•	His dashboards show combined North region performance across branches.

Scenario 5: Same Transporter, Different Permissions in Different Branches
The Situation
A transporter (trucking company) may work with multiple branches. But the relationship with each branch may be different — one branch may allow editing, while another only allows viewing.
Real-World Example
	•	Transporter “ABC Logistics” works with both Branch A (Pune) and Branch B (Nagpur).
	•	Pune’s team actively manages ABC’s trips and needs to edit transaction details.
	•	Nagpur’s team only monitors ABC’s deliveries and should not change anything.
How It Is Set Up
	•	Pune branch users tagged with ABC Logistics → Full access.
	•	Nagpur branch users tagged with ABC Logistics → View-only access.
What Should Happen
	•	Pune users can create, edit, and manage trips for ABC Logistics.
	•	Nagpur users can only view ABC Logistics trips.
	•	No accidental cross-editing happens between branches.
	•	Both teams see the same transporter but with different permission levels.

Scenario 6: Operations Team vs Finance Team
The Situation
Operations and Finance are two different departments that need to see the same data but for different purposes. Finance should never modify operational data, while Operations should work freely within their scope.
Real-World Example
	•	The Finance team at head office needs to review freight costs, invoices, and payment statuses across all journeys.
	•	The Operations team at each branch needs to create and manage journeys, assign vehicles, and update statuses.
	•	Finance should see everything but touch nothing. Operations should work within their assigned area only.
How It Is Set Up
	•	Finance users → View-only access across all branches and segments.
	•	Operations users → Full access, but only within their assigned branch and business segment.
What Should Happen
	•	A Finance user can open any journey from any branch and view all details including costs.
	•	A Finance user cannot edit journey status, reassign a vehicle, or delete a record.
	•	An Operations user can fully manage journeys but only within their own scope.
	•	This prevents accidental data changes by non-operational staff.

Scenario 7: Dashboard and Report Filtering Based on Access
The Situation
Users need to see charts, reports, and dashboards that reflect only their area of responsibility. A regional head should not see national numbers unless they have access.
Real-World Example
	•	Ms. Anita is the SPD North Regional Head. When she opens the performance dashboard, she should see only SPD North metrics.
	•	Mr. Raj is a Business Head. He should be able to compare SPD North vs SPD South performance side by side.
	•	A branch user tagged with FMCG should only see FMCG-related numbers on their dashboard.
How It Is Set Up
	•	Dashboards automatically filter data based on the user’s assigned tags.
	•	Users with multiple tags can toggle between views or see combined data.
What Should Happen
	•	Ms. Anita’s dashboard shows only SPD North data — delivery counts, on-time percentage, etc.
	•	Mr. Raj can see and compare data across all SPD regions.
	•	An FMCG-tagged user sees only FMCG metrics, not Cement or other segments.
	•	No user sees data outside their permitted scope on any report or chart.

Scenario 8: New Branch User with No Attributes Assigned Yet
The Situation
When a new user is added to the system, there may be a gap before their access tags are set up. The system needs a clear rule for what happens during this time.
Real-World Example
	•	A new employee, Sunil, joins the Pune branch on Monday.
	•	His manager creates his account but has not yet assigned him any access tags.
	•	Sunil needs to start working immediately, but his specific permissions are not defined yet.
How It Is Set Up
	•	The system offers two options, and each company can choose what works for them:
	•	Option A (Strict): Sunil sees nothing until tags are assigned. This is safer but may delay work.
	•	Option B (Open Start): Sunil gets full access to his branch, and restrictions are added later as tags are defined.
What Should Happen
	•	Under Option A: Sunil logs in and sees a message like “Access not configured. Contact your admin.”
	•	Under Option B: Sunil logs in and sees all Pune branch data, which gets narrowed down once his tags are set.
	•	The company’s admin decides which option to use during initial setup.
	•	The chosen option applies consistently across the entire organisation.

Scenario 9: Temporary Access for a Limited Time
The Situation
Sometimes a person needs access to data or systems that they do not normally have, but only for a short period — such as when covering for someone on leave or handling a seasonal workload.
Real-World Example
	•	Ms. Deepa is the South Regional Head. She goes on a two-week medical leave.
	•	Mr. Arvind, a senior operations executive, is asked to cover for her.
	•	He needs the same access Ms. Deepa has, but only for those two weeks. After that, his access should automatically go back to normal.
How It Is Set Up
	•	Mr. Arvind is given temporary South Region access with a start date and end date.
	•	The system tracks this as time-limited and auto-removes it when the period ends.
What Should Happen
	•	During the two weeks, Mr. Arvind can do everything Ms. Deepa normally does in the South region.
	•	On the day the temporary access expires, the system automatically revokes it.
	•	No admin action is needed to remove the access — it disappears on its own.
	•	An audit trail records that the access was temporary and why it was given.

Scenario 10: Access Based on Multiple Conditions at Once
The Situation
Sometimes a user’s access depends on more than one condition. For example, they may need to match a specific branch AND a specific product type AND a specific region — all at once.
Real-World Example
	•	Meera works at the SPD branch but only handles FMCG goods in the North region.
	•	She should not see Cement data in the North region, and she should not see FMCG data in the South region.
	•	Her access must match all three conditions: SPD branch + FMCG product + North region.
How It Is Set Up
	•	Meera is tagged with three labels: SPD branch, FMCG product type, and North region.
	•	The system checks all three tags together before showing any data.
	•	Only records that match ALL three conditions are visible to Meera.
What Should Happen
	•	Meera sees only FMCG journeys from the North region within SPD.
	•	She does not see Cement journeys in SPD North.
	•	She does not see FMCG journeys in SPD South or any other region.
	•	If any one condition does not match, the record is hidden from her.

Scenario 11: External Partner (Transporter or Vendor) Portal Access
The Situation
External parties like transporters, raw material suppliers or LSP may need limited access to the system — just enough to see their own indents, trips etc, without any visibility into internal operations or other partners’ data.
Real-World Example
	•	ABC Supplier is a third-party vendor company that provides raw materials for a shipper
	•	They log into a portal to see their assigned trips, update delivery status, and upload proof of delivery.
	•	They must not see trips assigned to other transporters, internal pricing, route strategies, or branch-level reports.
How It Is Set Up
	•	ABC Supplier is given a partner account tagged only with their set of routes, vehicles and materials
	•	The system restricts their view to only records where ABC Supplier is the assigned Vendor.
What Should Happen
	•	ABC Supplier sees only their own trips — pickup points, delivery destinations, and status updates.
	•	They cannot see trips belonging to XYZ Supplier or any other transporter.
	•	They cannot access internal company data other trips or any transactions which are not theirs
	•	They can update only the fields allowed (e.g., delivery status, proof of delivery).

Scenario 12: Emergency Access (Break-Glass Scenario)
The Situation
In urgent situations — like a shipment stuck at a checkpoint, a vehicle breakdown, or a time-sensitive delivery — someone outside the normal access scope may need to step in and take action immediately.
Real-World Example
	•	A high-value shipment is stuck at a toll plaza due to a documentation error.
	•	The person who normally handles this route is unreachable.
	•	Mr. Suresh, a senior manager from another region, needs to step in and update the journey record to clear the issue. He does not normally have access to this branch or region.
How It Is Set Up
	•	Mr. Suresh requests emergency access through the system.
	•	An authorised admin grants one-time or short-duration access with a reason logged.
	•	The system marks this as an emergency override.
What Should Happen
	•	Mr. Suresh gets temporary access to the specific journey or branch.
	•	He can make the necessary changes to resolve the issue.
	•	Once the emergency period ends, access is revoked automatically.
	•	The system records who granted access, when, why, and what actions were taken.
	•	This trail is available for audit and compliance review.

Scenario 13: Organisational Restructuring (Branch or Region Changes)
The Situation
When the company reorganises — such as merging branches, splitting regions, or moving areas from one head to another — the system must handle the transition smoothly without losing historical data.
Real-World Example
	•	SPD North region is being restructured. Three districts (Meerut, Agra, Mathura) that were under North are now moving to East.
	•	Ms. Priya (North Head) should no longer see those three districts’ new data.
	•	Mr. Kiran (East Head) should now see and manage those districts.
	•	But historical records from when those districts were under North should still be accessible for reporting.
How It Is Set Up
	•	The master data relevant to the three districts must be are moved from SPD_NORTH to SPD_EAST.
	•	Future journeys in those districts are automatically routed to East.
	•	Historical data retains its original tags for audit purposes, or both heads get read access to the transition period data.
What Should Happen
	•	Mr. Kiran immediately starts seeing new journeys from the transferred districts.
	•	Ms. Priya no longer sees new data from those districts.
	•	Historical reports still show the correct data for the period when those districts were under North.
	•	No data is lost or orphaned during the transition.

Scenario 14: Maker-Checker (Create and Approve Workflow) //Tech to check if this needs to be handled with ABAC or using approval engine
The Situation
For certain high-value or sensitive operations, one person should create the record and a different person should review and approve it before it takes effect. This prevents errors and fraud.
Real-World Example
	•	Ravi, a dispatch executive, creates a new journey entry for a high-value electronics shipment.
	•	Before the journey can be dispatched, it needs approval from his supervisor, Ms. Kavitha.
	•	Ravi should not be able to approve his own entries. Ms. Kavitha should be able to approve but not necessarily create journeys herself.
How It Is Set Up
	•	Ravi has “Create” permission for journeys.
	•	Ms. Kavitha has “Approve” permission for the same branch/segment.
	•	The system enforces that the creator and approver must be different people.
What Should Happen
	•	Ravi creates the journey. It goes into “Pending Approval” status.
	•	Ms. Kavitha reviews and either approves or sends it back for correction.
	•	Ravi cannot approve his own journey, even if he has a supervisor role elsewhere.
	•	Only after approval does the journey become active for dispatch.
	•	The system logs who created and who approved each record.

Scenario 15: Different Departments See Different Fields on the Same Record
The Situation
A single journey record may contain information relevant to multiple departments. But not every department should see every field. For example, finance should see costs, operations should see routes, and the customer should see only delivery status.
Real-World Example
	•	Journey #1234 has details like: route, vehicle, driver, freight cost, customer charges, margin, delivery ETA, and proof of delivery.
	•	The operations team needs to see route, vehicle, driver, and ETA.
	•	The finance team needs to see freight cost, customer charges, and margin.
	•	The customer portal should only show delivery ETA and proof of delivery.
How It Is Set Up
	•	Each role/department user is tagged with attributes which contain fields they can see.
	•	The system hides fields that a user does not have permission to view.
	•	This applies on screens, reports, and any exported data.
What Should Happen
	•	An operations user opens journey #1234 and sees route, vehicle, driver, ETA. Cost fields are hidden.
	•	A finance user opens the same journey and sees costs and margins. Route and driver details may be hidden or shown as read-only.
	•	A customer logging into the portal sees only ETA and delivery proof.
	•	No one sees more than what their role allows, even on the same record.

Scenario 16: Bulk Actions Across Mixed Access Boundaries
The Situation
When a user tries to update many records at once (bulk action), some records may fall within their access and some may not. The system must handle this gracefully.
Real-World Example
	•	Mr. Anil selects 200 journeys to update the status to “Completed.”
	•	Out of these, 150 are in his region (North), and 50 belong to the South region.
	•	He has full access to North but view only access to South.
How It Is Set Up
	•	The system checks each record against Mr. Anil’s access tags before applying the update.
What Should Happen
	•	The system updates the 150 North region journeys successfully.
	•	The 50 South region journeys are skipped.
	•	Mr. Anil receives a clear summary: “150 updated. 50 skipped (Read only access).”
	•	The system does not silently fail or block the entire operation because of the 50 inaccessible records.

Scenario 17: Auditor Access (Time-Scoped, Read-Only, Full Visibility)
The Situation
Internal or external auditors may need to review all data across the entire organisation, but only for a specific time period. They should see everything but change nothing.
Real-World Example
	•	An external auditor is reviewing all logistics operations for the financial year April 2024 – March 2025.
	•	They need to see every journey, every branch, every transporter, and every cost entry within that period.
	•	They must also see the action log — who created, edited, approved, or deleted each record.
	•	They should not be able to modify any data.
How It Is Set Up
	•	The auditor is given a special access profile: read-only, all branches, all segments, with a date range filter of April 2024 to March 2025.
	•	The profile includes access to system logs and audit trails.
What Should Happen
	•	The auditor can search and view all journeys within the specified period.
	•	They can see action logs showing who did what and when.
	•	They cannot edit, delete, or create any record.
	•	Data outside the date range is not visible to them.
	•	Once the audit engagement ends, their access is deactivated.

Scenario 18: Conflicting Access Rules — Which One Wins?
The Situation
A user might end up with two tags that conflict with each other. For example, one tag gives them full access to a region, and another tag says they can only view data in the same region. The system must have a clear rule for which one applies.
Real-World Example
	•	Ms. Lakshmi has two tags:
	•	Tag 1: SPD_NORTH → Full access (create, edit, delete).
	•	Tag 2: SPD_ALL → View-only access (as a business head).
	•	When she opens a North region journey, should she have full access or view-only?
How It Is Set Up
	•	The system must follow one of these pre-decided rules:
	•	Rule A (Most Permissive Wins): Full access wins because it is the broader permission.
	•	Rule B (Most Restrictive Wins): View-only wins because it is the safer default.
	•	Rule C (Explicit Priority): The admin sets which tag takes priority over others.
	•	The chosen rule is decided by the company during initial setup and applied consistently.
What Should Happen
	•	If Rule A: Ms. Lakshmi can edit North region journeys (full access wins).
	•	If Rule B: Ms. Lakshmi can only view North region journeys (stricter rule wins).
	•	If Rule C: Depends on what priority the admin has set for each tag.
	•	The system never leaves this ambiguous — it always resolves to a clear answer.
	•	Admins can review conflict resolution logic from the settings panel.

📦 Illustrative Example with Sample Data
User
Ms. Lakshmi | Tag 1: SPD_NORTH (Full CRUD) | Tag 2: SPD_ALL (View-Only)
Journey in Question
Journey #SPD-N-101 | Delhi → Jaipur | DL-01-AA-1111 | 10 MT ACC Cement | Rajdhani Carriers
This journey is in SPD North — both tags apply.
If Rule A (Most Permissive Wins)
✅ Ms. Lakshmi CAN edit Journey #SPD-N-101 because SPD_NORTH grants full access.
SPD_NORTH (Full CRUD) overrides SPD_ALL (View-Only) for North region data.
If Rule B (Most Restrictive Wins)
❌ Ms. Lakshmi can only VIEW Journey #SPD-N-101.
SPD_ALL (View-Only) overrides SPD_NORTH (Full CRUD) as the stricter rule.
If Rule C (Admin Priority)
Admin has set SPD_NORTH priority = 1 (highest), SPD_ALL priority = 2.
✅ SPD_NORTH wins → Ms. Lakshmi CAN edit North region journeys.

	•	Ms. Priya is the North Regional Head. Mr. Kiran is the South Regional Head.

	•	Ms. Priya does not receive any notification about it.

Scenario 20: Self-Service Access Delegation
The Situation
A branch head or team lead going offline (vacation, travel, illness) should be able to delegate their access to a trusted colleague for a defined period, without needing to contact an IT admin every time.
Real-World Example
	•	Mr. Amit, the Pune Branch Head, is going on a week-long trip with limited connectivity.
	•	He wants to delegate his access to Ms. Swati, his deputy, for that week.
	•	He does this himself from the system without raising a support ticket.
How It Is Set Up
	•	The system provides a “Delegate My Access” option for authorised roles.
	•	Mr. Amit selects Ms. Swati, sets the start and end dates, and confirms.
	•	The system grants Ms. Swati the same access Mr. Amit has, for the specified duration only.
What Should Happen
	•	Ms. Swati gets Mr. Amit’s access for exactly one week.
	•	After the week ends, her access automatically reverts to her original permissions.
	•	The system logs the delegation: who delegated, to whom, for how long, and why.
	•	Mr. Amit can cancel the delegation early if he returns sooner.

Scenario 22: Access Based on Journey Status or Stage
The Situation
Sometimes, access should change depending on what stage a journey is in. For example, a dispatcher can edit a journey while it is being planned, but once it is dispatched, only a supervisor should be able to make changes.
Real-World Example
	•	A journey goes through these stages: Planned → Dispatched → In Transit → Delivered → Closed.
	•	Ravi (dispatcher) can edit the journey while it is in “Planned” status.
	•	Once it moves to “Dispatched,” only his supervisor, Ms. Kavitha, can make changes.
	•	Once it is “Closed,” no one can edit it — it becomes a permanent record.
How It Is Set Up
	•	Permissions are linked to both user tags AND journey status.
	•	Dispatcher role: edit allowed only when status is “Planned.”
	•	Supervisor role: edit allowed until status is “Closed.”
	•	Closed journeys: read-only for everyone.
What Should Happen
	•	Ravi can modify journey details while the journey is being planned.
	•	Once dispatched, Ravi can view but not change the journey. Only Ms. Kavitha can.
	•	After the journey is marked “Closed,” neither Ravi nor Ms. Kavitha can edit it.
	•	This protects completed records from accidental or unauthorised changes.

Scenario 23: Access Revocation When an Employee Leaves or Transfers
The Situation
When someone leaves the company, moves to a different department, or is transferred to another branch, their old access must be removed promptly to prevent unauthorised use.
Real-World Example
	•	Rajesh was the Chennai Branch Head. He resigns and his last working day is Friday.
	•	By Saturday, he should have zero access to the system.
	•	Separately, Meena transfers from the North region to the South region. Her North access should end, and South access should begin.
How It Is Set Up
	•	HR or admin triggers an access change when an employee exits or transfers.
	•	The system supports immediate deactivation of accounts.
	•	For transfers, old tags are removed and new tags are added on the effective date.
What Should Happen
	•	After Friday, Rajesh cannot log in or access any data.
	•	Meena’s North region access ends, and South region access begins on the transfer date.
	•	Historical records still show Rajesh and Meena as the people who performed past actions (for audit purposes).
	•	The system logs the date and reason for every access change.
